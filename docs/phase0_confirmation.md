# Phase 0 范围确认记录

生成日期：2026-05-27  
状态：已根据用户反馈完成第一轮确认  
关联文档：`docs/rebuild_plan.md`

## 1. 原项目功能范围

确认结论：原项目功能清单中的数据域全部保留。

新项目目标不是压缩数据范围，而是在原项目基础上建设更丰富、更详实、更可维护的股票数据维护工程。以下数据域均进入新项目设计范围：

| 数据域 | Phase 0 结论 |
|---|---|
| A 股全市场股票池、证券主数据、交易日历 | 保留并增强 |
| 日线行情、复权因子、日度基础指标 | 保留并作为核心基础库 |
| 指数、行业、概念、板块映射 | 保留并增强 |
| 个股资金流、行业资金流、两融、北向资金 | 保留，缺失和源端限制显式管理 |
| 财务报表、财务指标、财务明细 | 保留并重点增强 |
| 样例导出、Parquet 出口、审计报告、变量标签 | 保留并制度化 |
| Agent/自动化更新入口 | 保留为薄封装 |

特别确认：

1. 财务明细设计需要继续展开，不能停留在原项目的 `financial_detail` 粗粒度设计。
2. 概念板块设计需要继续展开，不能只保留当前概念定义和静态映射。
3. 衍生变量设计需要继续展开，当前重建方案只是一级框架，不作为最终完整变量库。

## 2. 核心变量与衍生变量关系

用户反馈：新项目核心变量和衍生变量都需要保留；变量之间可以通过合适方式映射连接。

确认结论：

1. 新项目不采用“只保留核心变量、丢弃扩展变量”的方案。
2. 新项目会同时建设基础变量、核心衍生变量和扩展衍生变量。
3. 变量之间通过统一主键、变量注册表和视图/出口层连接，而不是把所有变量强行塞进一张不可维护的大表。

建议落地方式：

| 层级 | 保存方式 | 用途 |
|---|---|---|
| 基础变量 | 基础事实表和维表 | 数据维护、审计、复现 |
| 核心衍生变量 | 领域驱动物理表，例如 `derived_price_technical`、`derived_financial_quality` | 高频查询和日常维护 |
| 扩展衍生变量 | 可分模块物理表或扩展表 | 丰富变量库，按需出口 |
| 统一连接层 | `stock_features_core`、`stock_features_full` 等视图 | 让下游按需要选择核心或完整变量 |
| 变量元数据 | 变量注册表 | 管理中文名、口径、依赖、来源、缺失规则 |

后续设计原则：

- 核心和扩展不是二选一，而是维护优先级、刷新频率和出口视图的区别。
- 所有变量必须能通过 `(ts_code, trade_date)` 或明确的时点键进行连接。
- 财务、行业、概念、公司行为等低频变量必须通过 point-in-time 映射到日频，不允许未来函数。

## 3. 标签/目标变量

确认结论：标签/目标变量不纳入本工程。

原因：

1. 本工程定位为股票数据维护，不负责量化选股、回测训练和标签工程。
2. 未来收益、训练标签、模型目标等应由其他工程基于本工程维护的数据生成。
3. 第一阶段专注基础库、衍生变量库、更新机制、验证机制和文档同步机制。

处理方式：

- 不建设 `label_targets`。
- 不在默认视图中提供 `target_ret_*` 等未来收益字段。
- 如果为数据完整性审计需要保留少量未来窗口检查，也只能作为验证工具内部逻辑，不作为数据产品出口。

## 4. 日常刷新与历史修复边界

确认结论：

1. 最近 10 个交易日的历史修复可以默认触发。
2. 较远期历史数据应在基础库建设阶段尽量一次性建设完整。
3. 超过最近 10 个交易日的远期变量刷新，需要显式确认后再触发。

建议执行规则：

| 场景 | 默认行为 |
|---|---|
| 日常盘后更新 | 自动刷新最新已完成交易日 |
| 最近 10 个交易日源数据修复 | 默认允许 |
| 最近 10 个交易日衍生变量重算 | 默认允许 |
| 超过 10 个交易日的历史源数据修复 | 需要显式确认，除非处于基础库建设阶段 |
| 超过 10 个交易日的衍生变量刷新 | 需要显式确认 |
| 复权口径变化、公式变化、DDL 变化 | 需要显式确认并触发历史修复/全量重建 |
| 基础库初建或迁移验收 | 允许全量建设和全量校验 |

新项目应将该规则固化到 pipeline：

- `daily` 模式：自动处理最新交易日和最近 10 个交易日窗口。
- `history` 或 `repair --full` 模式：必须显式调用。
- 如果系统检测到需要超过 10 个交易日的影响范围，应输出计划并等待确认，而不是自动执行。

## 5. 工程定位

确认结论：本工程专注于股票数据维护。

明确不纳入本工程第一阶段的内容：

| 不纳入项 | 说明 |
|---|---|
| 量化选股策略 | 由其他工程完成 |
| 回测引擎 | 由其他工程完成 |
| 模型训练数据标签 | 由其他工程完成 |
| 策略组合、交易信号生产 | 由其他工程完成 |
| 人工看盘报表优先设计 | 非第一优先级，除非服务数据审计 |

本工程第一优先级：

1. 数据源接入和维护。
2. 基础库建设。
3. 衍生变量库建设。
4. 增量更新机制。
5. 数据质量审计。
6. 数据字典、变量字典和文档同步。
7. 稳定的数据出口，供其他工程使用。

## 6. Phase 0 后续展开项

根据本次确认，后续需要优先细化三块设计：

1. 财务明细数据设计：拆分分红、股东人数、十大股东、股本结构、主营构成、业绩预告、业绩快报、审计意见、商誉/减值、现金分红等子域。
2. 概念板块数据设计：概念定义、股票-概念映射历史、概念成分变化、概念行情、概念资金流、概念热度和多概念暴露。
3. 衍生变量完整设计：抛弃旧 A-J 字母框架，按领域驱动模块继续扩展变量清单、变量层级、刷新频率、依赖关系和验证规则。

## 7. Phase 0 结论

Phase 0 已确认：

- 数据范围：全部保留并增强。
- 变量范围：基础变量、核心衍生变量、扩展衍生变量都需要。
- 标签/目标变量：本工程不做。
- 日常刷新：最近 10 个交易日默认自动修复；更远历史需显式确认。
- 工程定位：只做股票数据维护，不做选股、回测、训练。

下一步建议进入 Phase 1 之前，先补充两份更细的设计文档：

1. `docs/data_contract.md`：基础表、财务明细、概念板块、数据源和主键约束。
2. `docs/variable_registry_design.md`：基础变量与领域驱动衍生变量注册表设计。
