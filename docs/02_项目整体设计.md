# 02 项目整体设计

更新日期：2026-06-05

本文是当前工程的总体设计入口，继承历史 `rebuild_plan.md`、Phase 2 确认记录和 Phase 3 各模块设计。本文重点说明项目如何从原工程重建为更清晰、更完整、更高效、更可审计的数据维护系统。

## 1. 设计目标

本工程的目标不是构造策略，也不是直接给出股票好坏判断，而是维护股票分析所需的事实数据层。

核心目标：

1. 全 A 股基础数据维护，含沪深京、北交所、退市股票历史数据。
2. 基础变量尽量丰富，覆盖行情、估值、财务、资金、指数、板块、公司行为、治理和元数据。
3. 衍生变量尽量完整，覆盖技术分析、财务分析、市场分析、截面分析和事实状态。
4. 所有低频数据遵守 point-in-time 原则，避免未来函数。
5. 核心物理表和完整视图分层，兼顾查询效率和字段丰富度。
6. 通过 schema registry、Excel 数据字典、审计报告和主文档建立同步机制。

## 2. 总体分层

| 层级 | 职责 | 当前状态 |
|---|---|---|
| source layer | Tushare/API 拉取、限流、重试、字段探测 | Phase 2 已建设主要基础数据 |
| raw/staging layer | 保存源返回快照、raw wide 表、`payload_json` 留痕 | 财务和事件表采用混合方案 |
| base layer | 清洗后的基础事实表和维表 | Phase 2 已完成主体基础库 |
| repair/asof layer | 可复现修复、公告日映射、历史补洞 | Phase 2/3 已用于财务和事件变量 |
| derived core layer | 高频使用、日增量必须维护的物理衍生表 | Phase 3 已完成 |
| derived full view layer | 宽周期、扩展字段、按需计算视图 | Phase 3 已完成主要模块 |
| metadata layer | schema、变量中文名、公式、口径、质量规则 | Excel 字典已建立，后续需持续维护 |
| validation layer | 完整性、PIT、主键、枚举、常量列、覆盖率审计 | Phase 3 已生成模块审计 |
| export layer | 统一出口视图、Parquet、样例工作簿 | Phase 4 优先建设 |

关键设计：

1. DuckDB 主库保存基础表、衍生核心表、视图和审计辅助对象。
2. `config/schema_registry.json` 记录表对象、字段、中文含义、衍生逻辑和口径。
3. Excel 数据字典由注册表和数据库事实生成，是人工审阅字段的主入口。
4. 历史 Markdown 字典只作为归档，不再作为主入口。

## 4. 基础变量总体设计

基础变量按数据域组织，而不是按脚本来源组织。

| 数据域 | 内容 | 设计重点 |
|---|---|---|
| 证券主数据 | 股票代码、名称、交易所、市场、上市/退市状态 | 全 A、退市股票、北交所同等维护 |
| 交易日历 | 交易日、前后交易日、是否开市 | 所有日频任务的时间基准 |
| 个股行情 | OHLCV、成交额、涨跌幅、停复牌、涨跌停价 | 原始交易事实口径 |
| 复权基础 | 复权因子、后复权/前复权价格 | 连续统计优先后复权，展示兼容可用前复权 |
| daily_basic | 估值、市值、股本、换手、量比 | 一等基础表，不作为临时修复表 |
| 财务报表 | 利润表、资产负债表、现金流量表 | raw wide + `payload_json` + 标准化视图 |
| 财务指标 | Tushare `fina_indicator` 完整指标 | 支撑 as-of、quality、growth |
| 财务事件 | 分红、预告、快报、审计、主营构成、回购 | 高价值事件拆结构化对象 |
| 资金交易 | 个股资金流、两融、北向、龙虎榜 | optional 域不阻塞主流程 |
| 行业概念 | 申万一级/二级行业、概念成分 | 多概念暴露，不只保留单一最强概念 |
| 指数市场 | 宽基、行业指数、指数行情、成分权重 | 指数成分权重按月维护 |
| 元数据审计 | 字段、表、更新时间、质量指标 | 驱动数据字典和质量报告 |

## 5. 财务 raw 存储设计

Phase 2 已确认采用混合方案：

1. 三大报表和 `fina_indicator` 建 raw wide 物理列。
2. raw wide 表同时保留 `payload_json`，作为源返回留痕和字段变更兜底。
3. 标准化财务表/视图使用稳定字段名，供 Phase 3 衍生变量使用。
4. 事件稀疏表采用核心物理列 + 可选 `payload_json`。

采用混合方案的原因：

1. 只保存 `payload_json` 虽然抗字段变化，但查询、审计、字典生成和衍生计算都低效。
2. 全部展开物理列查询效率高，但面对接口字段变化缺少兜底。
3. 混合方案兼顾完整性、可查询性、审计能力和未来兼容性。

## 6. 衍生变量总体设计

Phase 3 已抛弃旧 A-J 框架，采用领域驱动模块体系：

| 模块组 | 核心对象 | 说明 |
|---|---|---|
| 日频主干 | `derived_daily_spine` | 所有日频衍生变量的连接基准 |
| 交易技术 | price、volume、return、volatility、constraint 五组对象 | 技术分析和交易制度事实 |
| 估值规模 | `derived_valuation_size` | 估值、市值、股本和历史分位 |
| 财务 | `derived_financial_asof/quality/growth` | PIT 财务状态、质量和多周期成长 |
| 资金流 | `derived_capital_flow` 及缓存表 | 个股资金、两融、北向、龙虎榜事件 |
| 行业指数 | `derived_sector_concept_context`、`derived_index_market_context` | 行业、概念、指数、市场宽度 |
| 截面转换 | `derived_cross_sectional` | rank、percentile、z-score、中性化残差 |
| 公司行为 | `derived_corporate_action` | 分红、预告、快报、审计、主营、回购、解禁 |
| 持有人治理 | `derived_ownership_governance` | 质押、股东户数、十大股东、集中度 |
| 综合事实状态 | `derived_composite_state` | 多域事实状态和条件计数，不做评分 |

## 7. 核心物理表与完整视图策略

本工程采用“核心物理表 + 完整视图”的混合策略。

核心物理表适合：

1. 高频查询字段。
2. 日增量必须落库字段。
3. 下游常用字段。
4. 计算成本高或需要稳定审计的字段。

完整视图适合：

1. 周期枚举更丰富的字段。
2. 非核心但有分析价值的扩展字段。
3. 可通过 SQL 窗口函数按需计算的字段。
4. 暂不确定是否需要长期物化的字段。

这个策略已经用于交易技术、估值规模、财务成长、资金流、行业指数、截面转换、公司行为、持有人治理和综合事实状态。

## 8. 复权设计原则

复权不能一律处理，必须按业务场景判断。

| 场景 | 口径 |
|---|---|
| 连续收益、均线、动量、波动、回撤、ATR | 后复权 |
| 涨跌停、触板、一字板、停牌、成交事实 | 原始口径 |
| 成交量、成交额、换手率、量比 | 原始交易事实口径 |
| 市值、估值、股本 | Tushare 原始市值/估值口径 |
| 财务报表和财务指标 | PIT as-of，不复权 |
| 公司行为和治理事件 | 事件事实口径，不复权 |
| 截面转换 | 继承上游变量口径 |

## 9. 增量与效率设计

原项目日增量较慢，新项目设计上强调：

1. 日常任务基于交易日历和各表最新日期做 gap planning。
2. 近 10 个交易日默认修复源数据和衍生变量。
3. 超过 10 个交易日的远期历史刷新需要显式确认。
4. 衍生变量支持 read context window + write window，例如计算 120 日指标时读取足够历史窗口，但只写入目标刷新窗口。
5. 宽视图字段尽量不全部物化，降低主库空间占用。
6. optional 数据域如北向、龙虎榜、稀疏事件不阻塞主日批。
7. 每次刷新后生成审计报告和数据字典更新，避免悄悄漂移。

## 10. 文档同步机制

本项目明确要求文档和项目实际同步。

同步机制：

1. 字段变更先更新 schema registry 或变量注册配置。
2. 运行建表/建视图脚本。
3. 运行审计脚本验证对象存在、行数、列数、主键、枚举、PIT。
4. 重新生成 `global_variable_dictionary.xlsx`。
5. 更新主文档对应章节。
6. 历史草案和旧报告归档，不作为当前事实入口。

后续 Phase 4 建议把上述步骤封装为统一 CLI 和每日质量报告。
