# 07 运行维护与数据字典

更新日期：2026-06-07

## 1. 运行原则

1. 优先使用 WSL 环境执行工程任务。
2. 全量历史构建按模块执行。
3. 日增量刷新应按依赖顺序执行。
4. 近 10 个交易日历史修复可默认触发。
5. 超过 10 个交易日的历史修复需要显式确认。
6. 每次字段或模块更新后必须同步 schema、变量字典和 Excel 数据字典。
7. optional 数据域缺失不应阻塞主流程，但必须在审计报告中标记。
8. 每次 daily-light 正式补数前必须刷新证券主数据，确保新上市股票、退市状态变化和基础公司信息及时进入主库。

WSL 推荐命令入口：

```bash
cd "/mnt/d/Opencode Workspace/Stock_Maintainance"
source .venv-wsl/bin/activate
```

## 2. 推荐模块依赖顺序

```text
基础库
  -> derived_daily_spine
  -> trading technical modules
  -> valuation_size
  -> financial_asof / quality / growth
  -> capital_flow
  -> sector_concept_context / index_market_context
  -> cross_sectional
  -> corporate_action
  -> ownership_governance
  -> composite_state
  -> unified feature views
  -> audit reports
  -> global variable dictionary
```

其中：

1. `derived_daily_spine` 是所有日频模块的底座。
2. 财务、公司行为、治理必须依赖公告日或有效可得日。
3. 截面转换依赖前面各事实模块。
4. `composite_state` 依赖多个上游模块，因此放在最后。
5. Phase 4 的统一出口视图应在所有模块和审计完成后构建。

## 3. 增量刷新建议

日常增量刷新建议分为：

| 类型 | 刷新范围 | 是否默认触发 |
|---|---|---|
| 证券主数据 | 全 A 股上市/退市/暂停状态、公司信息、状态历史 | 是，必须先于日行情补数 |
| 日行情和 daily_basic | 最新缺口交易日至最新已完成交易日 | 是 |
| 近端修复 | 最近 10 个交易日 | 是 |
| 财务近期披露 | 最近若干公告日和报告期 | 是，需记录窗口 |
| 远期历史修复 | 超过 10 个交易日的历史区间 | 否，需要显式确认 |
| 全量重建 | 模块级或全库级 | 否，需要显式确认 |

daily-light 推荐顺序：

```bash
stock-maintain daily-light --as-of-date <today> --dry-run --output-prefix <run_id>
stock-maintain daily-light --as-of-date <today> --output-prefix <run_id>
```

`daily-light` 会按以下顺序编排执行：

```bash
stock-maintain sync-master --start-date 20060101
stock-maintain validate-daily --as-of-date <today> --output-prefix <run_id>_precheck
stock-maintain sync-daily-range <missing_start> <missing_end>
stock-maintain sync-market-behavior-range <missing_start> <missing_end>
stock-maintain sync-adj-factor-range <missing_start> <missing_end>
stock-maintain sync-index-daily <missing_start> <missing_end>
stock-maintain build-features --start-date <validation_or_missing_start> --end-date <missing_end>
stock-maintain create-views
stock-maintain validate-daily --as-of-date <today> --output-prefix <run_id>_postcheck
```

说明：

1. `sync-master` 会刷新 `stock_basic_info`、`stock_company_info`、`stock_status_history`、`trade_calendar` 和 `index_basic_info`，用于捕捉新股、退市和状态变化。
2. 新股上市后，主数据刷新应先引入 `ts_code` 和上市日期，再由后续日行情、复权因子、涨跌停和衍生构建自然补齐其日频记录。
3. 若新股上市日期早于当前缺口窗口，但日频数据未覆盖，应在 precheck 报告中显式暴露，并由人工决定是否执行超过 10 个交易日的历史修复。

周度复核推荐命令：

```bash
stock-maintain weekly-full --as-of-date <today> --dry-run --output-prefix <run_id>_weekly_plan
stock-maintain weekly-full --as-of-date <today> --output-prefix <run_id>_weekly_compare
```

说明：

1. `weekly-full` 默认使用最近 40 个开放交易日作为参照窗口、最近 10 个开放交易日作为请求比较窗口。
2. 若已有全量参照快照范围短于请求比较窗口，命令会自动取快照共同窗口与请求窗口的交集作为实际比较窗口。
3. 当前首版复用已有快照，不自动重建主库。真正的影子全量重建参照应在后续单独设计，避免污染生产表。

衍生变量增量应采用：

1. `write_window`：实际重写的交易日窗口。
2. `read_context_window`：为计算滚动指标读取的历史上下文窗口。
3. 审计窗口：检查写入窗口内行数、主键、缺失率和异常枚举。

例如计算 120 日均线时，可以读取 120+ 个交易日上下文，但只写入最近 10 个交易日。

## 4. 数据字典维护

当前主数据字典：

`outputs/variable_dictionary/global_variable_dictionary.xlsx`

生成脚本：

```bash
stock-maintain refresh-dictionary --output-prefix <run_id>_refresh_dictionary
```

若需要手工调用底层 Excel 生成器，可使用：

```powershell
& 'C:\Users\Administrator\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe' scripts\build_global_variable_dictionary.mjs
```

也可在 WSL 中调用项目可用的 Node/Python 环境执行对应脚本，最终产物仍写入上述路径。

维护要求：

1. 字段新增、删除、改名后必须重新运行数据字典生成脚本。
2. `Table_Index` 是全局入口。
3. `Table_Index` 中 Sheet 名配置跳转链接，不再单独生成“打开”列。
4. 物理表名应提供中文描述。
5. 基础变量和衍生变量均应有中文字段说明。
6. 衍生变量应写清楚来源表、来源字段和公式形式的衍生逻辑。
7. Excel 主文件被占用时，生成器可能输出时间戳副本；需要确认后替换主文件。

## 5. 文档维护机制

主文档维护要求：

1. 主目录只保留当前事实状态文件。
2. 历史文件统一归档到 `docs/99_archive/`。
3. 新增文档必须使用编号前缀。
4. 新增文档必须说明用途和继承关系。
5. 若替代旧文档，应在文档中说明替代关系。
6. 文档不能只保留摘要，应保留关键设计细节、边界、原因和对象清单。
7. 若文档与数据库事实不一致，应优先修正文档或重跑审计确认事实。

## 8. 后续工程化建议

Phase 4 应优先把当前人工步骤封装为稳定运行机制：

1. 统一 CLI：基础库刷新、衍生模块重建、审计、字典生成。
2. 每日质量报告：最新交易日、行数、缺失率、PIT、枚举、覆盖率。
3. 统一出口视图：`stock_features_core`、`stock_features_plus`、`stock_features_full`。
4. 文档漂移检查：数据库对象、schema registry、Excel 字典、主文档之间做一致性检查。
5. 历史修复审批机制：超过 10 个交易日的刷新必须记录确认和原因。
