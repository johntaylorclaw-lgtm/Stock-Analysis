## 2. 财务 raw 存储方案建议

### 2.1 方案 A：只保存 `payload_json`

做法：每条 Tushare 财报记录保存关键键字段，加一个 `payload_json` 字段保存完整原始返回。

优点：

1. 最抗 API 字段变化，Tushare 增删字段时表结构不需要立即迁移。
2. 初期开发快，能保证“先完整保存所有原始数据”。
3. 适合保留审计留痕和原始快照。
4. 对事件型、稀疏型接口很友好。

缺点：

1. 查询和分析不方便，每次使用字段都要 JSON 提取。
2. 字段类型约束弱，数值、日期、文本不容易在数据库层校验。
3. DuckDB 列式存储优势无法充分发挥，筛选某个财务字段时仍要解析 JSON。
4. 自动生成数据字典、逐列完整性审计、分布审计会变弱。
5. 后续衍生变量计算需要反复从 JSON 抽取字段，代码容易变散。

适用场景：

- 原始留痕层。
- 字段变化大、使用频率低的接口。
- 暂时不确定字段结构的补充数据。

### 2.2 方案 B：全部展开为数据表物理列

做法：将 Tushare 返回字段全部展开为 DuckDB 物理列，例如 `financial_income_raw` 直接包含 `income` 的 85 个字段。

优点：

1. 查询最方便，字段可直接 SQL 访问。
2. DuckDB 列式存储效率更高，适合宽表字段选择。
3. 字段类型、非空率、分布、异常值可以逐列审计。
4. 数据字典和变量注册表能精确到每个字段。
5. 后续构建标准化财务视图和衍生变量更直接。

缺点：

1. 初期 schema 更长，DDL 和字段映射维护成本更高。
2. Tushare 字段变化时需要 migration 或兼容逻辑。
3. 财务报表字段多，银行、券商、保险和普通企业字段差异大，很多字段天然低填充。
4. 如果直接手写 300 多个字段，容易出现字段名错误和文档漂移。

适用场景：

- 三大财务报表。
- `fina_indicator` 这类高频使用的财务指标表。
- 后续要参与衍生变量、质量审计和数据字典的核心字段。

### 2.3 我的建议：混合方案

建议采用“宽表物理列 + `payload_json` 原始留痕”的混合方案。

具体做法：

1. 对 `income`、`balancesheet`、`cashflow`、`fina_indicator` 建 raw wide 物理表，尽量展开 Tushare 返回字段。
2. raw wide 表同时保留 `payload_json`，作为原始返回留痕和字段变更兜底。
3. 在 raw wide 表之上建立标准化视图，例如 `financial_income`、`financial_balance`、`financial_cashflow`、`financial_indicator`。
4. 标准化视图使用稳定变量名，例如 `net_profit_attr_parent`、`cash_and_equivalents`、`cf_from_operating`。
5. 事件稀疏表，如 `forecast`、`express`、`dividend`、`fina_audit`、`fina_mainbz`、股东、质押、回购等，使用“核心物理列 + 可选 payload_json”的方式。
6. schema 不手工硬编码所有 raw 字段，而是在 Phase 2 初次 API 探测后生成字段清单，再人工审阅字段命名映射。

推荐原因：

1. 本项目定位是数据维护，数据应尽量丰富详实，不能只保存少数核心字段。
2. 财务报表是高价值基础数据，必须支持逐列审计、数据字典、质量检查和后续标准化映射。
3. 只用 `payload_json` 会让财务数据变成“保存了但不好用”，不符合长期维护目标。
4. 只用宽表又会损失原始留痕和字段变更弹性。
5. 混合方案兼顾完整性、可查询性、审计能力和未来兼容性。

最终建议：

Phase 2 采用混合方案。即：

- 三大报表和 `fina_indicator`：raw wide 物理列 + `payload_json`。
- 标准化财务表/视图：稳定字段名，供衍生变量和下游工程使用。
- 财务增强事件表：核心物理列 + `payload_json`，按接口逐步扩展。
